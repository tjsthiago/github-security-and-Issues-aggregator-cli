## Project Overview

This project is built with **Node.js** and **TypeScript**, following **Domain-Driven Design (DDD)** principles as defined by Eric Evans, Vaughn Vernon, Rodrigo Branas, Uncle Bob (Robert C. Martin), and Alistair Cockburn. All code must adhere to **Clean Code** practices and **SOLID** principles.

---

## Project Objective

A **Command Line Interface (CLI)** tool written in TypeScript, oriented around Domain-Driven Design principles. The project acts as a **Fetcher** and **Aggregator** to consolidate GitHub Issues and Dependabot Alerts from a private GitHub repository (e.g., `nome-do-owner/nome-do-repositorio`). The tool transforms raw data into a unified **Report** in Markdown format, facilitating vulnerability analysis and resolution SLA tracking.

## Ubiquitous Language

To ensure alignment between technical and business concerns, the project adopts the following domain vocabulary. All code — class names, method names, variables, and file names — must use these terms consistently.

| Term | Definition |
|---|---|
| **Repository** | The code and metadata container on GitHub. |
| **Issue** | General apontamentos or open security findings in the repository. |
| **Dependabot Alert** | Security notifications about vulnerable dependencies. |
| **Vulnerability** | The documented security flaw, classified by Severity. |
| **Report** | The `.md` artifact generated by the tool containing consolidated data. |
| **Sequential Index** | Auto-incremental number assigned to each record in the Report. |
| **Delimiter** | Visual marker (`---`) used to separate blocks of information. |
| **PAT (Personal Access Token)** | Access credential for authentication against the GitHub API. |
| **Fetcher** | Infrastructure component responsible for querying the GitHub API. |
| **Aggregator** | Application service responsible for processing and compiling the Report. |

## Domain Use Cases

The following use cases define the core behavior of the CLI tool:

1. **Fetch Issues**: Use a PAT to authenticate against the GitHub API and list all open Issues from the target Repository.
2. **Fetch Security Alerts**: Query and list all active Dependabot Alerts, capturing the Vulnerability context and its Severity.
3. **Aggregate Issues to Markdown**: Process the list of Issues and compile a structured Report. Each entry receives a Sequential Index in its title and is logically separated by the Delimiter, clearly demarcating where each Issue begins and ends.
4. **Aggregate Alerts to Markdown**: Process the list of Dependabot Alerts and compile a Markdown Report applying the same Sequential Index and Delimiter rules used for Issues.

## Architecture & Design Principles

### Domain-Driven Design

Follow DDD tactical patterns strictly:

- **Entities**: Objects with a unique identity (`id`) that persists over time. Behavior must be encapsulated inside the entity — never expose raw setters. Use factory methods or constructors to enforce invariants.
- **Value Objects**: Immutable objects with no identity. Equality is determined by their properties. Always validate input in the constructor. Examples: `Email`, `CPF`, `Money`, `DateRange`.
- **Aggregates**: A cluster of Entities and Value Objects with a single Aggregate Root that enforces consistency boundaries. External objects may only reference the root.
- **Domain Events**: Represent something meaningful that happened in the domain. Named in past tense (`OrderPlaced`, `UserRegistered`). Raised inside aggregates and dispatched by the application layer.
- **Domain Services**: Stateless logic that does not naturally belong to a single Entity or Value Object.
- **Repositories**: Defined as interfaces in `/domain/repositories`. Implementations live in `/infrastructure/repositories`. Repositories must deal with Aggregate Roots only.
- **Application Use Cases**: Each use case is a single class with an `execute()` method. Use cases orchestrate domain objects and repositories. They must not contain business logic — that belongs to the domain.

### SOLID Principles

- **S** — Single Responsibility: each class/module has one reason to change.
- **O** — Open/Closed: extend behavior via composition, not modification.
- **L** — Liskov Substitution: subtypes must honor contracts of their base types.
- **I** — Interface Segregation: prefer small, focused interfaces over large general ones.
- **D** — Dependency Inversion: depend on abstractions, never on concrete implementations in the domain or application layers.

### Clean Code

- Functions and methods must do one thing and do it well.
- Names must be intention-revealing: classes, methods, and variables must communicate their purpose without comments.
- Avoid magic numbers and magic strings — use named constants or Value Objects.
- Keep functions small. Prefer them under 20 lines.
- Avoid deep nesting; use early returns (guard clauses) as recommended by Alistair Cockburn's patterns.
- No dead code, no commented-out code.

## Coding Conventions

### TypeScript

- Enable `strict: true` in `tsconfig.json`.
- Prefer `type` over `interface` for DTOs and plain shapes; use `interface` for contracts and repository ports.
- Avoid `any`. Use `unknown` when the type is genuinely unknown and narrow it explicitly.
- Always type function return values explicitly.
- Use `readonly` for Value Object properties and any immutable data.
- Prefer `const` over `let`; never use `var`.

### No Lint

This project does not use ESLint or any linter. Code quality is enforced through:
- TypeScript strict mode
- Code reviews
- Test coverage requirements

### HTTP

- Use **native Node.js `fetch`** for all HTTP calls.
- Never import or install `axios`, `got`, `node-fetch`, `ky`, or any other HTTP client library.
- Wrap `fetch` calls inside infrastructure adapters (never call `fetch` directly in use cases or domain).

### Environment Variables

- All sensitive or environment-specific values (tokens, database URLs, API keys) must be stored in `.env`.
- The `.env` file **must never be committed** to version control (it must be listed in `.gitignore`).
- Provide a `.env.example` with all required keys and placeholder values for documentation.
- Load environment variables through a dedicated config module in `/src/infrastructure/config`.

## Testing

### Requirements

- All **Domain Entities**, **Value Objects**, and **infrastructure layer components** must have:
  - **Unit tests** testing behavior in isolation (no I/O, no real database).
  - **Integration tests** testing the actual integration with external dependencies.
- Use cases must have unit tests covering all happy paths and relevant edge cases.
- Aim for meaningful coverage, not just coverage metrics. Every test must assert real behavior.

### Framework

Use **Vitest** for all tests (unit and integration).

### Running Tests

```bash
# Run all unit tests
npm run test:unit

# Run all integration tests
npm run test:integration

# Run all tests
npm test
```

### Unit Tests

- Located in `/tests/unit`, mirroring the `/src` directory structure.
- Must run without any network, database, or filesystem I/O.
- Use in-memory fakes or mocks for all repository and infrastructure dependencies.
- Each test must be deterministic, isolated, and fast.

### Integration Tests

- Located in `/tests/integration`.
- Test infrastructure components against real external dependencies (e.g., real database, real HTTP endpoint).
- Use test containers or a dedicated test environment for isolation.
- Must clean up state after each test run.

### Test Naming Convention

```
describe('ClassName / feature', () => {
  it('should <expected behavior> when <condition>', () => { ... })
})
```

## Security Considerations

- Never hardcode credentials, API keys, or secrets in source code.
- Validate all external input at system boundaries (infrastructure layer) before passing data into the domain.
- Do not log sensitive data (passwords, tokens, personal identifiers).
- Sanitize error messages returned to clients — never expose stack traces or internal details.
